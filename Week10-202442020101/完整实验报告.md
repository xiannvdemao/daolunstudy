## 二、根因分析

### DELETE 性能瓶颈分析

1. **内存层面**
   - DELETE 需要修改 `table_data.rows` 向量，需要迭代匹配每一行
   - SELECT 只需读取，开销更低

2. **磁盘 IO 层面（最大瓶颈）**
   - **每次 DELETE 都会调用 `persist_table()` 写入整个表**
   - 测试场景：10,000 次 DELETE，每次写整个表（从 10,000 行→0 行）
   - 总写入量 ≈ (10,000+9,999+...+1) × 每行大小 ≈ **5000万行数据的磁盘写入**

3. **索引层面**
   - 当前 DELETE 没有利用索引快速定位行
   - 即使有索引，索引更新也需要额外开销

## 六、优化方案建议（可选）

### 优化方案 1：WAL（Write-Ahead Logging）+ 延迟写入

```rust
struct ExecutionEngine {
    dirty_tables: HashSet<String>, // 标记需要写入的表
    auto_flush_count: usize,       // 自动刷新阈值
}

fn execute_delete(&mut self, stmt: DeleteStatement) {
    // ... 删除逻辑 ...
    self.dirty_tables.insert(stmt.table); // 只标记，不立即写
    
    if self.dirty_tables.len() >= self.auto_flush_count {
        self.flush_dirty_tables(); // 批量刷新
    }
}
```

**预期效果：** DELETE QPS 从 86.52 → **10,000+**

### 优化方案 2：索引加速删除

在 DELETE 条件评估时利用索引快速定位行，避免全表扫描。
