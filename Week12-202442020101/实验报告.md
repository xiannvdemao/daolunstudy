
### 关键发现

SELECT QPS 是 DELETE 的 **466 倍**，是 UPDATE 的 **832 倍**，是 INSERT 的 **1,187 倍**。写操作（DELETE/UPDATE/INSERT）全部因为磁盘 I/O 瓶颈导致性能极低。

## 二、根因分析（步骤2）

### DELETE 性能瓶颈

| 瓶颈 | 根本原因 | 优化方向 |
|------|---------|---------|
| **磁盘 I/O** | 每次 DELETE 都调用 `persist_table` 写整个表到磁盘 | WAL / 延迟写入 |
| **全表扫描** | WHERE 条件没有利用索引，O(n) 遍历所有行 | 索引加速定位 |
| **数据量累积** | 10000 次独立写入，总写入量约 5000 万行 | 批量操作 |

### 核心代码路径分析

```
execute_delete()
  → table_data.rows.retain(|row| !evaluate_where(...))  // 全表扫描 O(n)
  → self.storage.persist_table(&stmt.table)             // 每次写整个表到磁盘
      → FileStorage::save_table()
          → serde_json::to_string_pretty()              // 序列化所有数据
          → std::fs::write()                            // 写入文件
```

## 四、Harness 治理闭环图（步骤4）

### 完整闭环流程

```
① Issue: "DELETE QPS 需从 79 → 10K+"
  │
  ▼
② AI/人写代码 → PR
  │
  ▼
③ Gate 检查
  BP1 ✅  BP2 ❌（QPS 不达标）
  │
  ▼
④ 分析根因 → 设计优化方案
  （WAL 延迟写入 / 索引加速 / 批量提交）
  │
  ▼
⑤ 重新 PR → Gate 通过 → 合并
  │
  ▼
⑥ 知识沉淀
  · 更新基准数据（79 → 64,896）
  · 记录 Pattern 到 GBrain
  · 评估：Gate 阈值需要调整吗？
  │
  └──→ 回到 ①（下一轮优化）
```

### 如果 Gate 不存在会怎样？

低性能代码（DELETE 79 QPS）会直接合并到主分支，生产环境用户感受到极慢的 DELETE 操作。性能问题无法被自动化拦截，只能等用户投诉后人工发现。

## 五、PDCA 自优化闭环分析（步骤5）

### PDCA 迭代表

| 迭代轮次 | Plan | Do | Check | Act |
|---------|------|-----|-------|-----|
| **第1轮** | DELETE ≥ 10,000 QPS | 实现基本 WAL 延迟写入：每次 DELETE 只标记脏表，定时刷盘 | QPS = 5,000 → **FAIL** | 分析：定时刷盘间隔太长；单线程写盘仍有瓶颈 |
| **第2轮** | 继续优化，突破 10,000 | 引入脏表批量提交 + 多线程异步 WAL：channel 异步写盘 + 批量合并 | QPS = 35,000 → 未达标 | 分析：WHERE 全表扫描是新瓶颈；需要索引加速 |
| **第3轮** | 突破瓶颈，达到 E-09 | 索引加速 DELETE 定位：利用 B+Tree 索引直接定位要删除的行 | QPS = 64,896 → **达标！** | 更新基准数据 + 沉淀知识到 GBrain |

### 关键洞察

- **第1轮瓶颈：** 即使延迟写盘，单线程 I/O 仍然是瓶颈
- **第2轮瓶颈：** 写盘优化后，CPU 计算（全表扫描）成为新瓶颈
- **第3轮突破：** 索引消除了 O(n) 全表扫描，变为 O(log n) 索引查找
- **核心思想：** 每轮优化解决一个瓶颈，Gate 检查提供客观反馈，PDCA 闭环驱动持续改进

## 六、Agent CI 编排设计（步骤6）

### 3-Agent 并行分析流程

| Agent | 分析任务 | 预期发现 |
|-------|---------|---------|
| **explore** | 分析 DELETE 语句的所有调用点，追踪执行路径 | 发现每次 DELETE 后都会调用 `persist_table` 写整个表到磁盘 |
| **librarian** | 查阅外部文档，验证 WAL/延迟写入等最佳实践 | 确认延迟写入是数据库领域的标准优化方案 |
| **oracle** | 评估性能风险，预测优化效果 | 预测：引入 WAL 后 QPS 可达 5,000+，加入索引后可达 64,000+ |

### 流程架构

```
PR 创建
    │
    ├──► explore Agent    → 代码调用点分析
    │
    ├──► librarian Agent  → 外部文档验证
    │
    └──► oracle Agent     → 性能风险评估
            │
            ▼
    收集结果 → 合成报告 → 通知开发者
```

## 七、Gate 联动流程（步骤7）

### 完整流程

```
push 代码 → Gitea Webhook → Nomad 调度 Job
  → BP1 静态检查 → BP2 集成检查 → BP3 风险检查
  → 结果上报 PR 评论 → 开发者收到通知
```

### 三层 Gate 说明

| Gate | 检查内容 | 通过条件 |
|------|---------|---------|
| **BP1 静态检查** | cargo build + fmt + clippy | 编译成功、格式规范、无警告 |
| **BP2 集成检查** | cargo test + QPS 基准测试 | 测试全通过、QPS ≥ 10,000 |
| **BP3 风险检查** | Agent 分析结果 + 最佳实践合规 | 无高风险变更 |

## 八、审计日志场景分析（步骤8）

### 场景：DELETE 路径增加审计日志同步写盘

| 维度 | 分析 |
|------|------|
| **QPS 影响** | 基线 64,000 → 加审计后预估 50~80 QPS（退化 99.87%） |

## 九、回归检测流程设计（步骤9）

### 设计决策

| 问题 | 方案 |
|------|------|
| 基准数据存哪里？ | `baseline.json` 文件，提交到仓库 |
| 什么时候触发检测？ | 每次 PR 自动触发 |
| 退化多少算 FAIL？ | ≤5% PASS / 5%~20% WARN / >20% FAIL |
| 退化后流程？ | CI 评论标注 → 创建 Issue → PR 打标签 |

### 流程图

```
1. PR 创建/推送 → 触发 CI
2. 运行性能测试（cargo test + HTTP 压测）
3. 对比 baseline.json，计算退化率 Δ
4. 判定 PASS / WARN / FAIL
5. FAIL → 自动创建 Issue + 性能报告
6. 开发者修复 → 重新提交 PR → 回到步骤1
```

### 退化率公式

```
Δ = (current_p99 - baseline_p99) / baseline_p99 × 100%
```
