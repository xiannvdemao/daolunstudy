DELETE 优化 PDCA 迭代表（206 → 64,896 QPS）

迭代轮次 Plan（计划） Do（执行） Check（检查） Act（改进）
第1轮 DELETE ≥ 10,000 QPS 实现基本 WAL 延迟写入：每次 DELETE 只标记脏表，定时刷盘 QPS = 5,000 → FAIL ❌ 分析：定时刷盘间隔太长导致内存占用高；单线程写盘仍有瓶颈
第2轮 继续优化，突破 10,000 QPS 引入脏表批量提交 + 多线程异步 WAL：脏表标记 + channel 异步写盘 + 批量合并 QPS = 35,000 → 未达标（但退化率 Δ = +16,600%，大幅改善） 分析：WHERE 条件全表扫描是新瓶颈；需要索引加速行定位
第3轮 突破瓶颈，达到 E-09 地板 索引加速 DELETE 定位：利用已有 B+Tree 索引直接定位要删除的行，避免全表扫描 QPS = 64,896 → 达标！✅ 更新基准数据 + 沉淀知识到 GBrain

关键洞察

  • 第1轮瓶颈：即使延迟写盘，单线程 I/O 仍然是瓶颈
  • 第2轮瓶颈：写盘优化后，CPU 计算（全表扫描）成为新瓶颈
  • 第3轮突破：索引消除了 O(n) 全表扫描，变为 O(log n) 索引查找
  • 核心思想：每轮优化解决一个瓶颈，Gate 检查提供客观反馈，PDCA 闭环驱动持续改进
