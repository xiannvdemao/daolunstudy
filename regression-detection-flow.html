回归检测流程设计 — Web API 项目

关键性能指标：/api/users 接口响应时间 ≤ 200ms

4.1 设计决策

1. 基准数据存哪里？

存储在 baseline.json 文件中，提交到仓库。
内容：{ "api/users": { "p99_ms": 180, "p50_ms": 45, "timestamp": "2026-06-06" } }
好处：版本可控，每次变更可追溯。

2. 什么时候触发检测？

每次 PR 都触发回归检测。
PR 创建时自动运行性能测试，对比 baseline.json。
每周额外运行一次完整基准测试，更新 baseline。

3. 退化多少算 FAIL？

与 SQLRustGo 保持一致：
≤ 5% → PASS（噪声范围内）
5% ~ 20% → WARN（PR 需要说明原因）
> 20% → FAIL（PR Blocked）

4. 检测到退化后的流程？

1. CI 自动在 PR 评论中标注退化详情
2. 生成性能对比报告（附火焰图）
3. 自动创建 Issue 并分配给 PR 作者
4. PR 标记为 "performance-regression" 标签

回归检测判定标准

退化程度 判定 动作 说明
≤ 5% PASS PR 正常合并 响应时间从 180ms 变为 189ms → 噪声范围内
5% ~ 20% WARN PR 需要说明原因 响应时间从 180ms 变为 210ms → 需要开发者确认是否可接受
> 20% FAIL PR Blocked 响应时间从 180ms 变为 350ms → 必须修复后才能合并

PDCA 闭环：Plan → Do → Check → Act

核心公式

退化率计算：
Δ = (current_p99 - baseline_p99) / baseline_p99 × 100%

示例：
基准 P99 = 180ms，当前 P99 = 189ms → Δ = 5% → PASS
基准 P99 = 180ms，当前 P99 = 210ms → Δ = 16.7% → WARN
基准 P99 = 180ms，当前 P99 = 350ms → Δ = 94.4% → FAIL
